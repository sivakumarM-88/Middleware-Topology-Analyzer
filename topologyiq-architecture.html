TopologyIQ — complete architecture design

Intelligent middleware topology management platform

Core thesis

Every middleware topology is a directed graph. TopologyIQ treats it as one — optimizing with graph algorithms, onboarding new apps with auto-generated naming and placement, and letting users converse with their topology through an AI agent. Same engine works for MQ, Kafka, or RabbitMQ.

System overview

The platform has three layers: a graph engine (the muscle), an AI brain (Claude), and three user-facing modes. All three modes share the same underlying graph representation and reasoning capabilities.

Layer 1 — graph engine (the core)

Abstract data model

Four entity types that map to any middleware:

TopologyNode

MQ: queue manager
Kafka: broker
RabbitMQ: virtual host

Fields: id, name, type, region, business_metadata (TRTC, PCI, neighborhood, line_of_business)

TopologyEdge

MQ: sender/receiver channel
Kafka: replication link
RabbitMQ: binding

Fields: source_node, target_node, edge_type, name (auto-generated from naming engine)

TopologyPort

MQ: queue (local/remote/alias/xmit)
Kafka: topic-partition
RabbitMQ: queue

Fields: node_id, name, direction (Local | Remote | Alias | Transmission), remote_target

TopologyClient

Any app producing or consuming messages

Fields: app_id, app_name, home_node_id, role (Producer | Consumer), business_metadata (hosting_type, app_disposition, criticality)

Naming convention engine

One of the most practical features. Given a producer app and consumer app, the engine generates all MQ object names deterministically. Derived from the pattern visible in the sample data:

Queue naming pattern: {PROD_APP}.{CONS_APP}.{SYSTEM}.{FUNCTION}.{SUFFIX} SUFFIX values: RQST (request), RESP (response), ACK (acknowledgment), DAT (data) Example — app PAYMENTS sends to app SETTLEMENT: Local queue: PAYMENTS.SETTLE.PAYHUB.TXN.DAT Remote queue: PAYMENTS.SETTLE.PAYHUB.TXN.DAT (on producer's QM) Alias: PAYMENTS.SETTLE.PAYHUB.TXN.DAT.XA21 (if needed) XMIT queue: SETTLE.{TARGET_QM_NAME} Channel naming pattern: Sender: {FROM_QM}.TO.{TO_QM} Receiver: {TO_QM}.FROM.{FROM_QM} Example: Sender: WL6EX2C.TO.WQ26 Receiver: WQ26.FROM.WL6EX2C

The naming engine is pluggable — organizations can supply their own convention as a template, and the engine applies it consistently. No more ad-hoc queue names.

Optimization pipeline (revised)

new Stage 0: graph discovery — Infer channels from remote_q_mgr + xmit_q pairs. Classify all ports (Local/Remote/Alias/Transmission). Build the edge set. This is where we go from flat CSV to graph.
revised Stage 1: constraint enforcement — 1 QM per app. Home QM election uses q_type + role to distinguish actual connectivity from data lineage references. Producer + non-alias queue = real connection. App_id on alias = just a routing reference.
unchanged Stage 2: dead object pruning — Remove orphan QMs (no clients, no active traffic), orphan queues (no producer AND no consumer), unused aliases, dead channels.
unchanged Stage 3: community detection (Louvain) — Run on undirected QM graph. Discovers natural clusters. Guard: skip hub-spoke for communities with fewer than 3 members.
revised Stage 4: hub election + spoke wiring — Hub = highest betweenness centrality weighted by business criticality (PCI, TRTC, payment-critical). Replace N*(N-1) mesh channels with N spoke channels to hub. Cross-community traffic routes hub-to-hub.
unchanged Stage 5: queue + channel rationalization — Apply naming engine. Wire remote → xmit → channel → local paths. Decide alias retention/elimination (explainer justifies). Output target CSV in same schema as input.

Complexity scorer

C = 1.0 * |nodes| + 2.0 * |edges| + 5.0 * avg_degree + 8.0 * max_fan_out + 10.0 * avg_path_length + 15.0 * cycle_count + 3.0 * orphan_objects + 4.0 * cross_community_edges + 20.0 * density Scored at every stage boundary. Waterfall shows each stage's contribution.

Layer 2 — AI brain (Claude)

Claude operates as three specialized agents, all sharing the same graph context:

Inventory agent

Discovers the as-is topology from CSV. Maps every producer-to-consumer flow. Flags anomalies: orphan objects, constraint violations, unusual patterns (e.g., app on 5 QMs). Builds the initial graph and annotates it with business metadata.

Optimizer agent

Runs the 6-stage pipeline. Records a DecisionRecord for every transformation: stage, action, subject, from/to, reason, evidence, complexity_delta. This decision log is the backbone of explainability. Also handles Mode 2 (onboarding) — when a new app arrives, runs placement algorithm and generates setup.

Conversational agent

The user-facing chatbot. Has the full graph + decision log in context. Handles queries, what-ifs, refinements, and explanations. Can modify the target topology based on user feedback and recompute complexity. Generates MQSC commands, design documents, and onboarding instructions on demand.

Layer 3 — three user-facing modes

Mode 1: optimize (batch transformation)

User uploads the as-is CSV (like the 20K-row dataset). The system processes it through all 6 pipeline stages and produces the target topology.

1
Upload CSV — Drag-and-drop or file picker. Preprocessor normalizes any CSV schema into the abstract model. Single flat table, multiple CSVs, whatever — the adapter handles it.
2
Inventory scan — System displays: X queue managers found, Y queues, Z apps, N constraint violations, M orphan objects. User reviews before proceeding. Side-by-side: as-is graph (force-directed) + summary metrics.
3
Run optimizer — Pipeline executes all 6 stages. Progress bar shows each stage. Complexity waterfall builds in real time. Decision log populates.
4
Review target — Side-by-side topology comparison. Target graph on right. Metrics show reduction. User can click any node/edge to see the decisions that affected it. Can switch to chat mode to ask questions or request refinements.
5
Export — Target CSV (same schema as input, automation-ready). Complexity report (PDF with waterfall). Design document (AI-generated). MQSC provisioning commands.

Mode 2: onboard new application

A new app needs to connect. Instead of a human architect spending days figuring out queue names and QM placement, the system does it in seconds.

1
User provides requirements — "App RISK_ENGINE (Consumer) needs to receive transaction data from app PPCSM (Producer)." Plus metadata: neighborhood (Wholesale Banking), hosting type (Private PaaS), PCI (No), TRTC (03= 4:01 to 11:59 Hours).
2
Placement algorithm runs — Where should RISK_ENGINE live?
• Option A: same QM as PPCSM (WL6ER2D) → local queues only, zero new channels, lowest complexity
• Option B: different QM in same community → 1 new spoke channel to hub
• Option C: different community → 1 new cross-community channel via hub-to-hub backbone
System recommends Option A with reasoning: "Same neighborhood, same hosting type, no PCI conflict, adds zero complexity."
3
Naming engine generates all objects —
Local queue: PPCSM.RISKENG.WBANK.TXN.DAT
No remote queue needed (same QM).
No channel needed (same QM).
Persistence: Yes (matches PPCSM's existing pattern).
TRTC: 03 (inherited from requirement).
4
Output setup instructions — Human-readable setup guide + MQSC commands + CSV rows to append to the target dataset. Complexity delta shown: "Adding RISK_ENGINE increases score by 2.0 (1 new queue, 1 new client). No structural change."
5
Graph updated — Target topology now includes RISK_ENGINE. All future queries and what-ifs reflect the updated state.

Mode 3: converse with topology

The most differentiated feature. The topology becomes a living thing you talk to. Here are the conversation categories the chatbot handles:

Query conversations

Where does app 8AFK live?
App 8AFK (Hdc - Hdxknvlr Kavk & Hncvympqf) is on QM WL6ER2D in the Wholesale Banking neighborhood. It has 3 local queues: ZEEWALA.ACK, ZEEWALA.DAT, and ARCNE.ACK. It communicates with app PPCSM (PP Arcne VMU) on the same QM — all local, no cross-QM channels needed.

What-if conversations

What happens if we retire QM WL6EX2C?
Retiring WL6EX2C impacts 2 apps (8A, OK) and 2 remote queues. Here's the migration plan:

App 8A (DFCeth, Producer) → migrate to WQ26. Create local queue on WQ26 for direct PUT. Remove remote queue + xmit queue from WL6EX2C. Delete channel WL6EX2C.TO.WQ26.

Net impact: -1 QM, -2 remote queues, -1 xmit queue, -1 channel pair. Complexity delta: -18.4. Want me to apply this change?

Refinement conversations

I don't want RISK_ENGINE on WL6ER2D. Put it on WQ26 instead.
Moving RISK_ENGINE to WQ26 means it's in a different community than its producer (PPCSM on WL6ER2D). I'll need to create:

• Remote queue on WQ26: PPCSM.RISKENG.WBANK.TXN.DAT
• XMIT queue: RISKENG.WL6ER2D
• Channel pair: WL6ER2D.TO.WQ26 (already exists), WQ26.FROM.WL6ER2D

Complexity increases by 8.6 vs the same-QM placement. The tradeoff: you get network isolation between producer and consumer. Proceed?

Explain conversations

Why did you pick WL6ER2D as the hub for Wholesale Banking?
Three reasons:

1. Highest betweenness centrality (0.42) in the community — it's already the natural routing center for 8 spoke QMs.
2. Hosts 2 PCI-compliant payment flows with TRTC 00 (0-30 min recovery) — hub needs to be the most reliable node.
3. Private PaaS hosting — better capacity headroom than the Mainframe-hosted alternatives.

The next candidate was WL6ER2F with centrality 0.31, but it only hosts non-PCI flows.

Generate conversations

Give me the MQSC commands to set up the RISK_ENGINE connection
Here are the MQSC commands for QM WL6ER2D:

DEFINE QLOCAL('PPCSM.RISKENG.WBANK.TXN.DAT') +
  DEFPSIST(YES) PUT(ENABLED) GET(ENABLED) +
  DESCR('PPCSM to RISK_ENGINE txn data') +
  CLUSTER('WBANK')

No remote queues, xmit queues, or channels needed — both apps are on the same QM.

Dashboard design

Four tabs across the top, consistent across all modes:

Overview
Topology explorer
Onboard app
Chat

Tab 1: overview

Metric cards (QMs, channels, apps, complexity score with delta). Complexity waterfall chart (score reduction by pipeline stage). Decision log stream (searchable, filterable by stage/action). As-is vs target summary.

Tab 2: topology explorer

Side-by-side D3 force-directed graphs. Left = as-is (hairball), right = target (hub-spoke). Nodes colored by community (Louvain cluster). Click any node to see its queues, apps, channels. Guided form: new app name, role (producer/consumer), target app to connect to, neighborhood, hosting type, PCI flag, TRTC class. Tab 4: chat

Full-screen chat interface. The AI agent has the complete graph in context. Supports all conversation types: query, what-if, refinement, explanation, generation. Every what-if and refinement updates the graph in real time. The topology explorer reflects changes immediately.
